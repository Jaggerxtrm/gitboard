Net leak +16 conn / 30s × ~15 projects → 1000+ at 53 min uptime → "max waiting connections reached" → bd CLI timeouts across all projects. Fix: try/finally with `await client.disconnect()`. Post-fix: connections drain to 1 within one cycle. |

## Code Changes

### Key modifications
- `apps/gitboard/src/api/server.ts` — `startServer` now passes `xtrmDb` in both positional args of `createApp` (`forge-bi35`). One-line fix for a regression that broke the entire beads UI.
- `apps/gitboard/src/dashboard/pages/console/Graph.tsx` — swap `beadsProjectName` → `beadsProjectId` (`forge-tyzt`). One-line fix for a graph emptiness regression.
- `apps/gitboard/src/server/observability/watcher.ts` — now `fs.watch` the `.db` + `.db-wal` + `.db-shm` sidecars and uses `max(mtime(.db), mtime(.db-wal))` as the change signal (`forge-0vuv`). 8.7s → 1.85s chip latency.
- `apps/gitboard/src/core/materializer/beads-adapter.ts` — `createLazyDoltClient.getIssues` wrapped in try/finally with `await client.disconnect()` (`forge-58ek`). Net connection drain from +16/30s to ~zero growth.

### New artifacts
- `docs/xtrm-observability-prd.md` — 441 lines / ~3500 words / 6 mermaid diagrams. Planning-ready PRD.
- `tools/probes/obs-materializer-lag.ts` — persisted probe script for re-running 0vuv timing measurements.
- `~/.local/bin/rotate-bd-dolt-log.sh` + `~/.config/systemd/user/bd-dolt-log-rotate.{service,timer}` — user-level log rotation (not in repo).

### Architectural decisions embedded in code
- Connection-lifetime ownership: any future `new DoltClient(...)` call in a hot path is now memory-pinned (`doltclient-disposal-hot-path-lesson-forge-58ek-2026`) — try/finally + disconnect, or refactor to a persistent pool.
- WAL-mode fs.watch contract (`materializer-obs-db-lag-root-cause-forge-0vuv`): any future fs.watch of a SQLite WAL-mode db MUST watch the `-wal` sidecar; main `.db` is a late signal.
